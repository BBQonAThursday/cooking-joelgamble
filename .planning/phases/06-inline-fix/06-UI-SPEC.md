# Phase 6 — UI Design Contract

> Visual and interaction contract for the Inline Fix / Categorize affordance. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase ships an icon-only pencil button on every grocery item row (`views/partials/grocery-item.njk`) and every recipe ingredient line (`views/recipe.njk`). Click expands a row-toggle inline editor: **Fix** (categories-only, for matched items) or **Categorize** (name + categories, for unmatched items). Save returns a full panel re-render driven by `HX-Current-URL`. All visual decisions extend the existing hand-rolled CSS vocabulary in `public/styles.css` — no new design system, no third-party components.

## Spacing Scale

Existing `public/styles.css` uses a near-8-point scale with one 12px exception already in production. Phase 6 adds nothing new — all editor markup uses these tokens.

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px | Pencil button inset; gap between badges |
| sm | 8px | Default control gap (matches `.grocery-item gap: 12px` minus the visual breathing the pencil adds) |
| md | 12px | Existing `.grocery-item gap` — pencil slots inside this gap, no change |
| md+ | 16px | `var(--gap)` — vertical rhythm between editor rows; padding-bottom on expanded `<li>` |
| lg | 24px | Editor's bottom margin when followed by another row |
| xl | 32px | Not used by Phase 6 |

**Pencil button hit area (D-68 verbatim — same dimensions as `.grocery-delete`):**
- `width: 24px; height: 24px;` (matches `.grocery-delete` line 314)
- `padding: 0;` (icon centered in box)
- Click target meets WCAG AA (≥24×24px); on touch devices the surrounding row gives ~44px effective tap area.

**Inline editor row (`.library-fix-editor`, `.library-categorize-editor`):**
- Replaces the `<li>` outer-HTML wholesale — keeps the `id="library-row-{{ entry.id }}"` (Fix) or `id="library-categorize-{{ surfaceItemId }}"` (Categorize) for HTMX outerHTML toggling per D-70.
- `padding: 12px 14px;` (matches `.grocery-item`)
- Internal form gap: `gap: 8px;` (matches existing `.library-add` form gap)

**Exceptions:** the existing 12px row padding is preserved (legacy from grocery-item.njk). Phase 6 does not introduce new exceptions.

## Layout

### Pencil placement

**Grocery item row (`views/partials/grocery-item.njk`):**

```
[✓ check]  text·························  [✏ pencil]  [× delete]
```

- Pencil is the second-to-last child, slotted between `.grocery-text` and `.grocery-delete`.
- Same width/height as `.grocery-delete` (24×24).
- The flex `gap: 12px` already on `.grocery-item` provides the breathing room — no additional margin needed.

**Recipe ingredient line (`views/recipe.njk`, `<li>` inside ingredient `<ul>`):**

```
text··················································  [✏ pencil]
```

- Pencil floats to the right of the line, vertically centered.
- The `<li>` becomes `display: flex; align-items: center; gap: 8px;`, with `.recipe-ingredient-text { flex: 1; }`.
- Pencil is `width: 20px; height: 20px;` here (slightly smaller than grocery-row pencil — 20px reads better against 16px body text without form chrome around it).

**Stable per-line id (Claude's-Discretion settled):** `id="recipe-ing-{{ recipe.id }}-{{ loop.index0 }}"` per the recipe-line `<li>`. Stable per render (deterministic input → output), unique per page, scopable for OOB if a future phase wants per-line swaps.

### Editor layout when expanded

**Fix editor (`library-fix-editor.njk`) — categories only:**

```
┌──────────────────────────────────────────────────┐
│ Library entry: garlic                  [Edit ↗]  │  <- header row (label + Edit-full-entry link)
│                                                  │
│ Recipe category   Grocery category               │  <- field labels (13px muted)
│ [Veg ▼]           [Produce ▼]                    │  <- two dropdowns side-by-side
│                                                  │
│ [Save]  [Cancel]                                 │  <- action row
└──────────────────────────────────────────────────┘
```

- Container: `<li class="library-fix-editor" id="library-fix-{surfaceItemId}">` — the surface-item id is the grocery `item.id` or the `recipe-ing-{recipe.id}-{idx}` id, NOT the library-entry id (so HTMX outerHTML correctly toggles back to the original row, NOT to a Library-tab row).
- **Two-column dropdowns** above 480px viewport; **stacked** below 480px (`flex-wrap: wrap; min-width: 140px;` per `<select>`).
- Save button reuses `.library-add button` styles (orange pill); Cancel reuses `.library-cancel` from `library-row-edit.njk` (transparent + bordered).
- "Edit full entry" link: text link in the header row, color `var(--accent)`, copy `Edit full entry →`. Target: `/library?q={encodeURIComponent(entry.name)}` (Claude's-Discretion default).

**Categorize editor (`library-categorize-editor.njk`) — name + categories:**

```
┌──────────────────────────────────────────────────┐
│ New library entry                                │  <- header (no name interpolation)
│                                                  │
│ Name                                             │
│ [garlic clove______________________]             │  <- pre-filled via normalizeIngredientText
│ ⚠ {inline error appears here on 400}             │
│                                                  │
│ Recipe category   Grocery category               │
│ [Veg ▼]           [Produce ▼]                    │  <- pre-selected via heuristic
│                                                  │
│ [Save]  [Cancel]                                 │
└──────────────────────────────────────────────────┘
```

- Container: `<li class="library-categorize-editor" id="library-categorize-{surfaceItemId}">`.
- Name input: full-width, 16px font (touch-friendly, matches `.library-add input`), `maxlength="200"`, `autocomplete="off"`, `required`, `autofocus`.
- Pre-fill values per D-76:
  - `name="{normalizeIngredientText(item.text)}"`
  - `recipeCategory` `<select>` has `selected` option matching `recipeCategoryOf(item.text)` (heuristic guess — works because D-76 says "no library param"; the heuristic falls through cleanly).
  - `groceryCategory` `<select>` has `selected` option matching `groceryCategoryOf(item.text)`.
- NO aliases input (D-76 — keeps scope tight; Library-tab full-editor handles aliases later).
- Inline error slot (`<div class="library-categorize-error">`) appears below the name input only when present (HTTP 400 conflict path).

### Viewport behavior

- **≥640px:** Two-column dropdown layout in both editors.
- **<640px (mobile):** Dropdowns stack; Save/Cancel buttons remain side-by-side (small enough to fit). The `@media (max-width: 640px)` block already exists in `public/styles.css` (line 413) — Phase 6 appends the editor stack-on-narrow rule there.

## Copywriting Contract

All toast strings are ASCII-only, verb-only literal strings — locked by D-78 and CLAUDE.md HTTP-header ASCII rule.

| Element | Copy |
|---------|------|
| Pencil aria-label (matched item) | `Fix categorization for {entry.name}` (interpolated server-side; `{entry.name}` is library data which CAN contain non-ASCII — but `aria-label` is HTML attribute, NOT HTTP header, so non-ASCII is safe here per CLAUDE.md scope) |
| Pencil aria-label (unmatched item) | `Categorize {item.text}` (HTML attribute, same safety) |
| Fix editor header | `Library entry: {entry.name}` (label-weight, muted; metadata only) |
| Categorize editor header | `New library entry` (literal; no interpolation) |
| Recipe-category field label | `Recipe category` |
| Grocery-category field label | `Grocery category` |
| Name field label (Categorize) | `Name` |
| Save button label | `Save` (matches Phase 5 `library-row-edit.njk` line 19) |
| Cancel button label | `Cancel` (matches Phase 5) |
| Edit-full-entry link | `Edit full entry →` (Fix editor only — directs to Library tab per FIX-03) |
| Primary CTA (this phase) | `Save` — fires `POST /library/:id/categories` (Fix) or `POST /library` (Categorize) |
| Save toast — Fix path | `Saved categories` (D-78, ASCII verb-only) |
| Save toast — Categorize path | `Added entry` (D-78, reused from Phase 5 D-67) |
| Empty state | NOT applicable in this phase — Phase 6 adds an affordance ON existing rows; the empty-grocery-list and no-ingredients states are unchanged from Phase 5. |
| Error state — Categorize name conflict (HTTP 400) | `Name "{typed}" is already used by entry "{owning entry name}". Open it in the Library tab.` (inline `<div class="library-categorize-error">` — silent toast per D-77 / Phase 5 D-61) |
| Error state — invalid category enum (HTTP 400) | Server returns the same editor fragment with the offending `<select>` reset to its previous value; no inline error text needed (the dropdown enum constraint makes user-visible error rare; toast silent). |
| 404 — entry not found | Plain-text body `Not found`, no toast (matches Phase 5 D-67 silent-404 pattern). |
| Destructive confirmation | NOT applicable in this phase — Phase 6 has no destructive action. Library-entry deletion remains a Library-tab-only affordance per FIX-03. |

**HTTP-header ASCII rule applied:** Toast strings (`Saved categories`, `Added entry`) are pure literal ASCII bytes. The inline error message DOES interpolate user-typed name and entry name — but it appears in the HTML response BODY (not the `X-Status-Toast` header), so non-ASCII is safe. This is the same pattern Phase 5 D-61 uses for the alias-conflict inline error.

## Interaction Contract

### Open editor (matched / Fix)

1. User clicks pencil on a grocery row or recipe ingredient line.
2. HTMX fires `hx-get="/library/{libraryEntryId}/categories-edit"` with `hx-target="closest li"` and `hx-swap="outerHTML"`.
3. Server returns `library-fix-editor.njk` rendered with the entry's current values pre-selected; the `<li>` outer wrapper carries `id="library-fix-{surfaceItemId}"` and `class="library-fix-editor"`.
4. HTMX outerHTML-swaps the original row with the editor.
5. Focus moves to the first dropdown (`autofocus` on `name="recipeCategory"` `<select>`).

### Open editor (unmatched / Categorize)

1. User clicks pencil on a row whose item has `libraryEntryId === null`.
2. HTMX fires `hx-get="/library/categorize-edit?text={item.text|urlencode}"` (encoded server-side via `{{ item.text | urlencode }}`).
3. Server returns `library-categorize-editor.njk` with name pre-filled (D-76 — `normalizeIngredientText`) and dropdowns pre-selected from heuristic; same outerHTML swap.
4. Focus moves to the name input (`autofocus` on `name="name"`).

### Save (both editors)

1. User clicks Save.
2. HTMX fires `hx-post` (Fix: `/library/:id/categories`, Categorize: `/library`) with `hx-target="closest li"` (or `body` for full panel re-render — see below) and the `HX-Current-URL` header automatically attached.
3. Server reads `HX-Current-URL`, picks the OOB target (`#grocery-list`, `#recipe-ingredient-groups-{recipe.id}`, or `#library-panel` per D-73), and returns:
   - **Primary swap target:** Empty body (or whitespace) — the editor `<li>` is removed from the DOM by the OOB panel re-render that replaces the entire surrounding list.
   - **OOB fragment:** `<{surface-panel} hx-swap-oob="true">...</{surface-panel}>` containing the freshly-rebuilt panel via `buildGroceryView` / `buildView` / `buildLibraryView` per surface.
4. Toast fires with verb-only ASCII string.
5. The item now appears in its new category group (or the entry exists, if Categorize). The pencil is again visible on the new row.

### Cancel (both editors)

**Settled from Claude's-Discretion (option B):** Client-side reset via a per-surface lightweight `GET` endpoint that returns the original row markup.

- Fix editor Cancel button: `hx-get="/library/cancel-fix?surface={grocery|recipe}&itemId={surfaceItemId}"` returning the read-only `<li>` fragment for that specific surface (pulls from `state` via `buildGroceryView` or `buildView`).
- Categorize editor Cancel button: same endpoint, `text=` query param so the server can rebuild the original row from item id + text.
- `hx-target="closest li"`, `hx-swap="outerHTML"`.
- No state change, no toast (`hx-headers='{"X-Skip-Toast": "1"}'` if needed — but a GET request with no `X-Status-Toast` header in the response will simply not fire a toast).

**Rationale:** Re-fetching is server-authoritative (always reflects current state) and avoids client-side serialization of the original markup. The endpoint is cheap (one `storage.get()` + one render). Planner may consolidate the two surface variants into one endpoint with a `surface` query param.

### 400 conflict (Categorize only)

1. User types a name that conflicts with an existing entry name OR an existing alias.
2. `POST /library` returns HTTP 400 with body = the same `library-categorize-editor.njk` fragment, with `categorizeError` set to the conflict message and the user's typed values preserved on the form.
3. The HTMX 4xx-swap meta tag (Plan 05-01) auto-swaps the body into the editor `<li>` via outerHTML.
4. Form stays open; toast remains silent (the inline error IS the user feedback).
5. Mirrors Phase 5 D-61 verbatim.
