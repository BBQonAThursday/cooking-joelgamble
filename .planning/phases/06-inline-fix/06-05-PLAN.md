<objective>
Wave 4: integration tests proving the click-edit-save round-trip works end-to-end across all three surfaces, plus the FIX-04 invariants on both grocery and recipe pages.

This is the final wave. After Plan 04 ships the route handlers, this plan adds tests that simulate the user flow:
1. **Round-trip per surface:** seed an item; GET the surface; assert pencil renders; POST the categories; re-GET the surface; assert the item moved to the new category group.
2. **FIX-04 invariants:** seed a library entry whose alias matches an item's text; rename the entry's `name`; assert the grocery row / recipe ingredient line STILL renders the original item text (not the canonical name). This is the most important regression test for Phase 6 — it locks in the contract from CLAUDE.md "Templates always display ingredient.text" and from REQUIREMENTS.md FIX-04.
3. **FIX-03 categories-only invariant:** assert the Fix editor fragment has zero `<input name="name">` and zero `<input name="aliases">` elements. (Already partially covered in Plan 02 GET tests; reinforced here as an end-to-end assertion.)
4. **Categorize SC#3 round-trip:** a previously-unmatched grocery item gets a Categorize editor; submit; the new entry exists in `state.library`; the next GET /grocery has the item now reporting `libraryEntryId` (i.e., the pencil now targets `/library/:id/categories-edit`, not `/library/categorize-edit`).

Purpose: Plans 01-04 cover individual route + template behaviors. This plan locks in the cross-route contract — that the user's mental model (click → fix → see fixed list) actually holds end-to-end. Without these tests, a regression in any single component (e.g., `decorateIngredients` losing its library threading, `buildGroceryView` losing `libraryEntryId` attachment) would not be caught at the unit level.

Output: ~10 new tests across 3 test files. No production code changes.
</objective>
