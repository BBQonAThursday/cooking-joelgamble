## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → GET /library | `req.query.q` and `req.query.filter` arrive as untrusted strings. They flow into `buildLibraryView` and are interpolated into HTML attributes (search input value, filter button hrefs, empty-state copy). Mitigation: Nunjucks `autoescape: true` (server.js line 9) escapes attribute and text contexts. |
| `state.recipes[].ingredients[]` → `findEntryInIndex` | Existing untrusted strings (scraped from external recipe URLs); already normalized in Phase 2/3. No new exposure. |
| `state.library[].name` / `aliases[]` → HTML rendering | Already-normalized strings, but may legitimately contain non-ASCII. Mitigated by Nunjucks autoescape; HTML attributes (hx-confirm) safely accept Unicode. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02-01 | Tampering (V5 Input Validation) | GET /library?q=&filter= query params | mitigate | `typeof req.query.q === 'string' ? req.query.q : ''` and same for `filter` — coerces non-string (e.g., array shapes from `?q=a&q=b`) to safe defaults. Filter values are NOT validated against an enum here; an unknown filter (e.g., `?filter=hax`) falls through to `All` behavior since neither Uncurated nor Unused matches — semantically safe (no crash, no privilege escalation). |
| T-05-02-02 | Tampering (XSS via entry name in HTML) | views/partials/library-row.njk `{{ entry.name }}` | mitigate | Nunjucks `autoescape: true` (server.js line 9) escapes `<`, `>`, `&`, `"` — entry names containing HTML chars render as text. | T-05-02-03 | Tampering (XSS via deleteConfirm in hx-confirm attribute) | views/partials/library-row.njk `hx-confirm="{{ entry.deleteConfirm }}"` | mitigate | Nunjucks attribute escaping covers `"` (the attribute boundary) and `&`.
