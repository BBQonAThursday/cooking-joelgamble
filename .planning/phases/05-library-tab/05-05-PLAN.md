Edit `routes/library.js`. Add the DELETE handler BEFORE `module.exports = router;` (placement: AFTER `router.post('/library/:id', ...)` from Plan 04).

```javascript
// LIB-06 / D-64..D-67: Delete entry. Removes from state.library; NEVER mutates state.recipes.
// Success response: empty body for the row position (HTMX outerHTML removes the row)
// + OOB footer fragment with updated unusedCount.
// 404: plain text (existing convention).
router.delete('/library/:id', (req, res) => {
  const id = req.params.id;
  const state = storage.get();
  const library = Array.isArray(state.library) ? state.library : [];
  const idx = library.findIndex(e => e.id === id);
  if (idx === -1) return res.status(404).type('text').send('Not found');

  // Remove the entry. CRITICAL invariant (LIB-06): do NOT touch state.recipes.
  // Categorization for any recipe ingredient strings that previously matched
  // this entry's aliases will fall back to the heuristic on the next render
  // (lib/calc.js#decorateIngredients + buildGroceryView already handle missing
  // library entries via the existing fallback path).
  library.splice(idx, 1);
  state.library = library;
  storage.save();

  // D-67: generic toast. Do NOT interpolate the deleted entry's name (HTTP header ASCII safety).
  setToast(res, 'Removed entry');

  // Compound response: empty primary fragment (row removal) + OOB footer.
  // The empty primary fragment is sent as a literal empty string before the footer;
  // HTMX outerHTML on #library-row-:id replaces the element with nothing, removing it.
  const updatedView = buildLibraryView(state);
  const footerHtml = injectOob(renderSync(req, 'partials/library-footer.njk', updatedView));
  res.type('html').send(footerHtml);
});
```

Key implementation points:

1. **`library.splice(idx, 1)` mutates the SAME array reference held by `state.library`**, then we reassign `state.library = library` defensively (no-op since it's the same reference, but explicit). `storage.save()` persists. This pattern matches the existing `lib/grocery.js#removeItem` mutation style.

2. **DO NOT touch `state.recipes`.** The LIB-06 invariant is that recipe ingredient strings are never mutated. The Plan 03 categorization-fallback path in `lib/calc.js` (existing — Phase 3 work) already handles "no library match" by falling back to the heuristic keyword tables. Verified by Plan 02 unit tests covering the empty-library buildLibraryView case.

3. **The response body is JUST the OOB footer** — no row-replacement fragment. When HTMX receives an empty/short response and `hx-swap="outerHTML"` targeting `#library-row-:id`, it tries to replace the element with whatever's in the body. If the body is empty (or contains only OOB fragments), HTMX removes the target element. The OOB footer fragment is processed separately via `hx-swap-oob="true"` injected by `injectOob`. Verified by RESEARCH §Compound Row + Footer Response and HTMX 2.0.4 docs.

4. **Edge case — empty library after deletion:** if `library.splice` leaves `state.library === []`, `buildLibraryView(state).unusedCount` returns 0, and the footer template renders an empty `<div id="library-footer" class="library-footer"></div>`. The OOB swap still updates the DOM (the footer element stays in place with empty content) — no crash.

5. **Verb choice — DELETE not POST.** Per RESEARCH.md "HTTP verb for delete" recommendation and per the existing `routes/grocery.js#delete` and `routes/recipes.js#delete` precedent. CONTEXT.md D-67 Claude's-Discretion explicitly authorizes either; the planner-final choice (per orchestrator's wave_plan_guidance) is `DELETE /library/:id` for parity with existing routes. The button in `views/partials/library-row.njk` (Plan 02) already uses `hx-delete="/library/{{ entry.id }}"` — that aligns with this verb.

DO NOT add a "POST /library/:id/delete" alias route. DO NOT add a confirmation page (hx-confirm handles it client-side). DO NOT add bulk-delete (out of scope per REQUIREMENTS.md).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → DELETE /library/:id | The :id path param is untrusted; flows into find-by-id (read) and splice (write). |
| state.library mutation surface | This is the only Phase 5 endpoint that REMOVES an entry. The recipes-untouched invariant (LIB-06) is the security-sensitive part — accidentally mutating recipes would corrupt user data. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05-01 | Tampering (V5 Input Validation — id) | DELETE /library/:id path param | mitigate | `findIndex(e => e.id === id)`; non-matching ids return 404. The id is never used as a regex source, eval target, or path concatenation source. |
| T-05-05-02 | Tampering (recipes integrity invariant) | DELETE handler scope | mitigate | Code never references `state.recipes` for write. Test #2 is the regression gate — any future change that mutates recipes during delete will fail this test. |
| T-05-05-03 | Tampering (HTTP header injection — toast) | setToast(res, 'Removed entry') | mitigate | Literal string; no interpolation. setToast strips `\r\n` defensively. |
| T-05-05-04 | Information Disclosure | 404 body 'Not found' | accept | Plain text, no leakage of which ids exist. |
| T-05-05-05 | Tampering (XSS via footer) | renderSync(library-footer.njk) | mitigate | Footer template only interpolates `unusedCount` (a number from buildLibraryView) — no user-controlled strings. |
| T-05-05-06 | Denial of Service | unbounded delete loops | accept | Per-request single splice; O(n) over library. No recursion. |
| T-05-05-07 | Repudiation (no audit log) | DELETE handler | accept | Out of scope per REQUIREMENTS.md. Single-user trust model; no audit trail. |
</threat_model>
