```javascript
// LIB-05 / D-60..D-63: Edit save. Validates name + categories + alias conflicts.
// Success: row fragment + OOB-footer fragment.
// 400: edit-form fragment with inline error (form stays open per D-61).
// 404: plain text (id not found).
router.post('/library/:id', (req, res) => {
  const id = req.params.id;
  const state = storage.get();
  const library = Array.isArray(state.library) ? state.library : [];
  const idx = library.findIndex(e => e.id === id);
  if (idx === -1) return res.status(404).type('text').send('Not found');

  const existing = library[idx];
  const body = req.body || {};
  const name = typeof body.name === 'string' ? body.name.trim() : '';
  const aliasesRaw = typeof body.aliases === 'string' ? body.aliases : '';
  const recipeCategory = typeof body.recipeCategory === 'string' ? body.recipeCategory : '';
  const groceryCategory = typeof body.groceryCategory === 'string' ? body.groceryCategory : '';

  // Parse aliases (D-60): split on ',', trim, drop empty, dedupe via Set.
  const aliases = [...new Set(
    aliasesRaw.split(',').map(a => a.trim()).filter(Boolean)
  )];

  // Helper: build the edit-form view object that re-renders the form with the
  // user's typed values + an inline error. This is used by all 400 paths.
  function renderEditFormError(errorMsg) {
    const formView = {
      id,
      name,
      aliases,
      aliasesDisplay: aliases.join(', '),
      recipeCategory,
      groceryCategory,
      curated: !!existing.curated,
      aliasError: errorMsg
    };
    const html = renderSync(req, 'partials/library-row-edit.njk', {
      entry: formView,
      RECIPE_CATEGORIES,
      GROCERY_CATEGORIES
    });
    // D-61: NO setToast on 400. The inline error is the user feedback.
    return res.status(400).type('html').send(html);
  }

  // Validation 1: name required.
  if (!name) {
    return renderEditFormError('Name is required.');
  }
  // Validation 2: recipeCategory enum.
  if (!RECIPE_CATEGORIES.includes(recipeCategory)) {
    return renderEditFormError(`Invalid recipe category '${recipeCategory}'.`);
  }
  // Validation 3: groceryCategory enum.
  if (!GROCERY_CATEGORIES.includes(groceryCategory)) {
    return renderEditFormError(`Invalid grocery category '${groceryCategory}'.`);
  }
  // Validation 4: alias conflict — pass excludingId so editing one's own alias is not self-conflict.
  for (const alias of aliases) {
    const conflict = aliasConflict(state, alias, id);
    if (conflict) {
      return renderEditFormError(
        `Alias '${alias}' is already used by '${conflict.name}'.`
      );
    }
  }

  // All valid: update in place. LIB-05 sets curated:true on save (regardless of prior value).
  library[idx] = {
    ...existing,
    name,
    aliases,
    recipeCategory,
    groceryCategory,
    curated: true
  };
  state.library = library;
  storage.save();

  // D-63: Compound response — primary row fragment + OOB-footer fragment.
  // CRITICAL: do NOT use respondWithUpdates — it would inject hx-swap-oob on the
  // row, breaking the primary swap target.
  setToast(res, 'Saved entry');
  const updatedView = buildLibraryView(state);
  const updatedEntry = updatedView.entries.find(e => e.id === id);
  const rowHtml = renderSync(req, 'partials/library-row.njk', {
    entry: updatedEntry,
    RECIPE_CATEGORIES,
    GROCERY_CATEGORIES
  });
  const footerHtml = injectOob(renderSync(req, 'partials/library-footer.njk', updatedView));
  res.type('html').send(rowHtml + '\n' + footerHtml);
});
```

Critical points (re-state the constraints to anchor against pitfalls):

1. **Compound response, not `respondWithUpdates`.** The row is the primary swap target (HTMX outerHTML targets `#library-row-:id` based on the form's `hx-target`). The footer is the only OOB fragment. Using `respondWithUpdates` would wrap the row in `injectOob`, which gives the row a redundant `hx-swap-oob` attribute and breaks the primary swap (RESEARCH.md Pitfall 2 / 05-PATTERNS.md "CRITICAL").

2. **400 returns the edit-form fragment, NOT the row.** D-61 says the form stays open on conflict. The fragment is rendered with `entry.aliasError` set so the inline `<div class="library-alias-error">{{ entry.aliasError }}</div>` from `library-row-edit.njk` appears under the aliases input.

3. **No `setToast` on 400.** D-61 explicitly silences the toast on conflict — the inline error is more persistent and contextual. The Plan-01 htmx-config meta-tag's omission of `error:true` for code 400 is what suppresses the existing layout.njk `showStatus('Error', true)` listener.

4. **`aliasConflict(state, alias, id)` — pass id as excludingId.** Editing one's own alias must not self-conflict. If the existing entry has alias `garlic` and the user submits the form unchanged, `aliasConflict` must return falsy.

5. **`curated: true` on every save.** Even if the entry was previously curated:true, the spread + override re-sets it. This is intentional per LIB-05 ("sets curated:true when an uncurated entry is edited" — re-setting the same value on already-curated entries is a no-op).

6. **The ELS pattern: `{ ...existing, name, aliases, recipeCategory, groceryCategory, curated: true }`** preserves any future fields (e.g., `createdAt`, future `nutrition` field per CLAUDE.md extensibility note) without enumerating them.

7. **`renderEditFormError` reconstructs the form view from the user's submitted values** (NOT from the existing entry's stored values). This is what "preserves typed values on error" means in D-61.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → POST /library/:id | Untrusted form submission with all four entry fields. The :id path param is also untrusted but used only for find-by-id (no SQL/no eval). |
| HTMX 400 swap path | The 400 response body is HTML — Nunjucks autoescape applies. The htmx-config meta tag (Plan 01) opts the response into being swapped into the DOM. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04-01 | Tampering (V5 Input Validation — name) | POST /library/:id name field | mitigate | `typeof === 'string' ? body.name.trim() : ''`; reject empty with 400 + edit-form fragment. Form maxlength=200 is soft cap; server enforces non-empty. |
| T-05-04-02 | Tampering (V5 Input Validation — categories) | POST /library/:id recipeCategory + groceryCategory | mitigate | Server-side enum check before write. Rejects with 400 + edit-form fragment AND inline error referencing the rejected value. |
| T-05-04-03 | Tampering (V5 Input Validation — aliases) | POST /library/:id aliases field | mitigate | maxlength=1000 on form. Server splits, trims, dedupes via Set. Each alias passed to `aliasConflict(state, alias, id)`. |
| T-05-04-04 | Tampering (alias-conflict bypass via own-id collision) | aliasConflict(state, alias, id) | mitigate | `excludingId` arg ensures editing one's own alias is not a self-conflict. Verified by Test #3 (no-op self-edit). |
| T-05-04-05 | Tampering (curated:true forced regardless of input) | save spread `curated: true` | mitigate | Server hardcodes `curated: true` — the form does NOT submit a `curated` field. User cannot un-curate via this route. (Un-curating is not in scope per LIB-05; the only path back to curated:false is auto-extract creating a new entry, never via edit.) |
| T-05-04-06 | Tampering (XSS via aliasError in 400 response) | renderEditFormError | mitigate | `aliasError` flows into the row-edit template's `{{ entry.aliasError }}` text node — Nunjucks autoescape escapes `<`, `>`, `&`. | T-05-04-08 | Information Disclosure | 404 response body | accept | `'Not found'` plain text — no information about whether `:id` exists in some other state. |
| T-05-04-09 | Denial of Service | aliasConflict per-alias loop | accept | Worst case: maxlength=1000 / 2 char min alias = 500 aliases × O(library) lookup. At 200-entry library: 100K comparisons, still <50ms. Acceptable. |
| T-05-04-10 | Repudiation | (no audit log) | accept | Out of scope per REQUIREMENTS.md "Out of Scope (v1) — no audit trail". Single-user trust model. |
