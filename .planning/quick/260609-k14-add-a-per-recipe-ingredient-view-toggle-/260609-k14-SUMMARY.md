# Phase quick-260609-k14 Plan 01: Add per-recipe ingredient view toggle Summary

**One-liner:** Per-recipe toggle persists viewMode ('original'|'processed') in state, flipping between flat scraped list and decorateIngredients categorized view via POST /recipes/:id/view returning body fragment + OOB toggle button.

## What Was Built

A toggle button on each recipe detail page lets users switch the Ingredients section between two views:

- **Original** (default): flat `<ul>` of ingredients in scraped order, no category headers, no pencil buttons — the pristine source ingredient list.
- **Categorized (Processed)**: the existing `decorateIngredients` grouped output with category headers and pencil controls.

The chosen view is persisted per recipe as `recipe.viewMode` in `state.json`. The toggle fires `POST /recipes/:id/view` which flips the value, saves, and returns an HTML fragment (body innerHTML) + OOB button update in one response — no full page reload.

## Key Implementation Notes

### CLAUDE.md render-time categorization honored
`decorateIngredients` is called only when `viewMode === 'processed'`. The original mode sets `ingredientGroups: []` and renders directly from `recipe.ingredients` (plain strings). The `viewMode` string itself is the only value stored -- never computed categories.

### Library OOB swap compatibility preserved
The outer `<section class="recipe-ingredients" id="recipe-ingredient-groups-{{ recipe.id }}">` is untouched. The new structure lives inside it: a `recipe-ingredients-head` div (h2 + toggle) and a `recipe-ingredients-body-{id}` div (swappable body). The library Save OOB path (`recipe-ingredient-groups-oob.njk`) continues to target the section id correctly. Pencils are only visible in processed mode, so the library OOB swap only fires when the categorized view is active -- acceptable pre-existing behavior documented in the plan.

### Existing tests updated
6 existing tests that expected category headers or pencil buttons on recipes without an explicit `viewMode` were updated to seed `viewMode: 'processed'`. These tests were asserting behavior that required the categorized view; the updates reflect the correct new default ('original').
